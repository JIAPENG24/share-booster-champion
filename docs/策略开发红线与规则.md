# 策略开发红线与规则

> 依据赛事技术合规要求、赛场规则、框架架构约束综合整理。

---

## 🔴 一、技术合规红线（不可触碰）

触犯以下规则可能导致**取消参赛资格**。

| # | 红线 | 说明 |
|---|------|------|
| TC-01 | 仅使用官方公开 API | 只允许使用官方文档化的 ROS topic、Agent API、SDK 接口 |
| TC-02 | 禁止访问仿真内部接口 | 不得连接 websocket、HTTP、Unix socket、共享内存等内部控制接口 |
| TC-03 | 禁止伪造控制命令 | 不得移动球、移动/重置机器人、暂停/关闭仿真、修改碰撞属性等 |
| TC-04 | 禁止连接 GameController 管理接口 | 不得伪造裁判/操作员/主办方控制请求 |
| TC-05 | 禁止端口扫描 | 不得枚举 localhost / Docker 网关 / host 网络 |
| TC-06 | 禁止读取非公开数据 | 不得访问容器/宿主机/其他进程的文件、环境变量、token、日志、配置 |
| TC-07 | 禁止读取系统元数据 | 不得读取 /proc、容器元数据、Docker 网络信息 |
| TC-08 | 禁止伪造通信数据 | 不得监听/注入/伪造 ROS/DDS/websocket/UDP 数据（官方接口除外） |
| TC-09 | 禁止干扰对手或裁判 | 不得干扰对方 agent、主办方服务、仿真进程、裁判进程或网络通信 |

## 🔴 二、框架架构红线（不可变动）

以下为行为树与系统架构层面的硬约束，**不得绕过或修改**。

| # | 红线 | 说明 | 来源 |
|---|------|------|------|
| AR-01 | 不可绕过 SafetyOverrides | 硬件状态兜底层，必须在 CommitTeamCommands 之前执行，策略不可跳过 | bt_structure.md |
| AR-02 | 不可绕过 DataLayer | 黑板是每帧唯一数据源，策略层只读黑板，不得直接对接 ROS 源 | bt_structure.md |
| AR-03 | 不可绕过 CommitTeamCommands | 行为树到执行器的唯一握手点，所有命令必须经此提交 | bt_structure.md |
| AR-04 | play/ 层不可导入 py_trees | 策略层保持框架无关性，不依赖 py_trees 库 | 入门示例策略 |
| AR-05 | 策略代码不可直接调用 boosteros | 只能输出 RobotCommand，由 runtime 通过 TeamRobotManager 统一调度 | developer_protocol.md |
| AR-06 | 不可移除 Player 守卫分支 | 每个 Player(N) 的 KickoffHold / PenaltyAvoid / PenaltyStopped 守卫必须保留 | bt_structure.md |
| AR-07 | 不可移除 WaitForBall 兜底 | 角色分配失败或球丢失时的安全回退节点 | bt_structure.md |
| AR-08 | 不可绕过 SafetyGuards | 全局安全守卫（NoGameStop / AllInactiveStop / StoppedPlayStop 等），执行优先级高于 PlayingPhase | bt_structure.md |
| AR-09 | SafetyOverrides 内三层守卫不可绕过 | AllowedGuard（罚下检查）、FallDownGuard（摔倒恢复）、WalkModeGuard（模式切换） | bt_structure.md |
| AR-10 | set_velocity 与 SoccerKickManager 互斥 | 同一机器人同一时刻只能使用其一，不可同时控制 | developer_protocol.md |

## 🔴 三、坐标与配置红线

| # | 红线 | 说明 |
|---|------|------|
| CR-01 | 不可硬编码 team_id | 需从环境变量 `SOCCER_TEAM_ID` 读取 |
| CR-02 | 不可硬编码 player_id 映射 | 需从环境变量 `SOCCER_ROBOT_NAMES` 读取，不假定 robot1/2/3 |
| CR-03 | 遵循己方坐标系约定 | 己方球门 x=-7.0，对方球门 x=+7.0，+x 为进攻方向 |
| CR-04 | 不可在策略层二次镜像坐标 | PlayContext 数据已归一化为己方视角，无需按 team_id 翻转 |
| CR-05 | 不可硬编码场地参数 | 场地尺寸、球门宽度、中圈半径等应读取配置，而非硬编码 |

## 🔴 四、赛场规则硬约束（策略代码必须遵守）

在策略行为实现中，必须确保以下规则不被违反。

### 4.1 站位约束

| # | 约束 | 适用阶段 | 阈值 |
|---|------|---------|------|
| RP-01 | 非开球方不得进入中圈 | set + kick-off | 中圈半径 1.5m |
| RP-02 | 非开球方不得越半场 | set + kick-off | x=0 |
| RP-03 | 开球方在对方半场中圈内 ≤1 人 | set + kick-off | — |
| RP-04 | 防守方距球 ≥1.45m（定位球球权未开放时） | set play | 1.45m |
| RP-05 | 不得出场外 | set | 场地 14m×9m |

### 4.2 运动约束

| # | 约束 | 适用阶段 | 阈值 |
|---|------|---------|------|
| RM-01 | Set 阶段路径 ≤0.15m（宽限 1s 后） | set | 0.15m |
| RM-02 | Stop 阶段路径 ≤0.3m（宽限 2s 后） | playing + stopped | 0.3m |
| RM-03 | 倒地检测：upDot ≤0.45 持续 ≤10s | ready/set/playing | 10s |
| RM-04 | 长时间无活动：球距 ≤3m，≤10s 几乎不动 | playing（非 stopped） | 10s |

### 4.3 定位球约束

| # | 约束 | 阈值 |
|---|------|------|
| RS-01 | kick-off 开球时限 | 10s |
| RS-02 | 其余定位球开球时限 | 45s |
| RS-03 | 超时后球权开放，双方可抢球，进球有效 | — |
| RS-04 | kick-off 直接进球需 ≥2 次触球或 set play 已过期 | — |
| RS-05 | 定位球乌龙（触球 ≤1 次且未过期）→ 不加分，判对方角球 | — |

### 4.4 避让豁免条件

当防守方同时满足以下条件之一时，可豁免避让 1.45m 限制：

| 豁免 | 条件 |
|------|------|
| 正在远离 | 500ms 窗口内径向速度 > 0.05m/s（远离球方向） |
| 门线防守 | 站在本方球门线上，y 在门宽 2.6m 范围内 |

## 🟢 五、允许修改的开发范围

### 5.1 推荐修改（主要）

| 层面 | 文件 / 目录 | 可修改内容 |
|------|------------|-----------|
| **团队战术** | `play/playbook.py` | 角色分配逻辑 `assign_roles()`、追球手选择 `select_chaser()`、新增战术模式 |
| **球员角色** | `play/default_roles.py` | 踢球目标 `kick_target()`、支援站位 `target()`、角色行为树构造 |
| **角色基类** | `play/role.py` | 新增角色派生类（如自定义 defender/interceptor） |
| **射门决策** | `tactics/targeting/` | 射门目标评分、传球评分、追球手评分、接应站位计算 |
| **导航避障** | `tactics/navigation.py` | 障碍物建模、避障算法 |
| **运动控制** | `tactics/motion.py` | READY 入场分段控制、行走控制参数 |
| **几何工具** | `tactics/geometry.py` | 坐标系转换、场地钳位 |
| **踢球迟滞** | `tactics/kick_hysteresis.py` | 进入/退出踢球范围阈值（默认：进入 2.5m，退出 3.0m） |
| **READY 站位** | `tactics/ready_stance.py` | 开球前站位方案 |
| **配置调优** | `soccer_framework/config.py` | `SoccerStrategyTuning`：移动速度、踢球力度、避障距离、避让距离等 |

### 5.2 谨慎修改（需理解框架后再调整）

| 层面 | 文件 / 目录 | 说明 |
|------|------------|------|
| **PlayingPhase 子树** | `behavior_tree/play_subtree.py` | 可调整 PlayingPhase 内的策略控制流，但需保证角色树结构完整 |
| **行为树动作节点** | `behavior_tree/nodes/actions.py` | 可扩展新的行为树动作叶节点 |
| **行为树条件节点** | `behavior_tree/nodes/conditions.py` | 可扩展新的条件判断节点 |

### 5.3 开发顺序建议

参考官方建议，按以下顺序由浅入深：

1. **调参优化** → 修改 `config.py` 中的 `SoccerStrategyTuning` 参数
2. **角色重塑** → 修改 `play/default_roles.py` 调整 Chaser/Supporter/Goalkeeper 行为
3. **战术调整** → 修改 `play/playbook.py` 调整角色分配策略
4. **架构升级** → 调整 `behavior_tree/` 控制流（需充分理解框架后操作）

## 🟡 六、框架底层（通常不改）

以下层为核心基础设施，通常无需修改。若确需修改必须具备充分理由。

| 文件 / 目录 | 职责 | 备注 |
|------------|------|------|
| `soccer_framework/types.py` | 核心数据类型 | 平台通信契约 |
| `soccer_framework/game_state.py` | GameController JSON 解析 | 裁判数据解析 |
| `soccer_framework/game_controller.py` | ROS GameController 适配器 | 裁判数据订阅 |
| `soccer_framework/ros_truth.py` | RosTruthProvider / PlayContextProvider | ROS 真值订阅与过滤 |
| `soccer_framework/robot.py` | RobotClient / TeamRobotManager | 机器人控制封装 |
| `soccer_framework/config.py` 中的 `SoccerConfig` | 全局配置基类 | 但 `SoccerStrategyTuning` 可改 |
| `behavior_tree/tree.py` | 顶层行为树创建 `create_team_tree()` | 组装所有子系统 |
| `behavior_tree/blackboard.py` | 黑板配置 | 数据交换中心 |
| `runtime.py` | 系统装配与控制循环 | 入口与生命周期管理 |
